Tercero · Feature Research

Team Task Management

A ground-up look at replacing the existing sticky-note task system with a proper role-aware, assignable task engine.

Prepared for Chris Austin · July 2026 · Operations module

01 Where We Are Today

The current "Tasks & Reminders" feature is built on a table named client_notes — a holdover from when tasks and notes were the same concept. It was later split into separate Notes and Tasks sections, but the underlying table stayed.

What it is now
  • Table named client_notes — confusing name
  • No assignee — tasks belong to whoever created them
  • No team awareness — everyone sees everything
  • No concept of "assigned by" vs "assigned to"
  • Members and owners are treated identically
  • No priority field
  • Statuses: TODO, DONE, ARCHIVED only
What we're building
  • Clean tasks table — purpose-built name
  • First-class assignee with role-aware assignment rules
  • Visibility scoped by assignment (personal vs team)
  • Separate created_by and assigned_to
  • Owner/admin can assign; members can only receive
  • Priority field: Low / Normal / High / Urgent
  • Adds IN_PROGRESS for active work tracking

What the existing UI already does well

The TasksAndReminders.jsx page is good. It has a grid view with date grouping (Overdue / Upcoming / Done / Archived), a kanban board with drag-and-drop between columns, client filter, search, and status tabs. None of this gets scrapped — we enhance it.

Recommendation: new table, keep the UI. The client_notes schema can't grow into a team task system without breaking changes. A fresh tasks table with the right columns from day one is cleaner than bolting columns onto a mislabeled table. The page, dialogs, and kanban board are worth keeping and extending.

02 New Data Model

The new tasks table separates the creator from the assignee, adds role-aware workspace scoping, and carries enough metadata for a proper team work queue.

CREATE TABLE tasks (
  id              UUID          PRIMARY KEY DEFAULT gen_random_uuid(),
  workspace_id    UUID          NOT NULL REFERENCES auth.users(id),
  -- who made it and who owns the work
  created_by      UUID          NOT NULL REFERENCES auth.users(id),
  assigned_to     UUID          REFERENCES auth.users(id),  -- NULL = personal task

  title           TEXT          NOT NULL,
  description     TEXT,

  status          TEXT          NOT NULL DEFAULT 'TODO'
                  CHECK (status IN ('TODO', 'IN_PROGRESS', 'DONE', 'ARCHIVED')),
  priority        TEXT          NOT NULL DEFAULT 'NORMAL'
                  CHECK (priority IN ('LOW', 'NORMAL', 'HIGH', 'URGENT')),

  due_at          TIMESTAMPTZ,
  completed_at    TIMESTAMPTZ,

  -- optional context links
  client_id       UUID          REFERENCES clients(id) ON DELETE SET NULL,
  campaign_id     UUID          REFERENCES campaigns(id) ON DELETE SET NULL,

  created_at      TIMESTAMPTZ   NOT NULL DEFAULT now(),
  updated_at      TIMESTAMPTZ   NOT NULL DEFAULT now()
);
Column Type Purpose
workspace_idUUID= get_my_agency_user_id(). All RLS is scoped here.
created_byUUIDThe user who created the task. Never changes.
assigned_toUUID?NULL = personal task (private to creator). Set = team task (assignee + all admins can see it).
statusTEXTTODO → IN_PROGRESS → DONE → ARCHIVED
priorityTEXTLOW / NORMAL / HIGH / URGENT. Affects sort order and visual treatment.
completed_atTIMESTAMPTZ?Stamped automatically when status moves to DONE. Useful for analytics.
client_idUUID?Optional link — task appears in the client's context.
campaign_idUUID?Optional link — task appears in the campaign's context.

Migration from client_notes

Existing records in client_notes migrate as personal tasks: created_by = user_id, assigned_to = NULL, workspace_id = get_my_agency_user_id(). Status mapping: TODO→TODO, DONE→DONE, ARCHIVED→ARCHIVED. Priority defaults to NORMAL. After migration, client_notes can be dropped.

03 Assignment Rules

Assignment follows the authority chain established by RBAC. You don't assign upward — you delegate downward.

Role Can assign to… Can be assigned by… Can create personal tasks?
Owner Admins, Members Nobody Yes
Admin Other Admins, Members Owner only Yes
Member Cannot assign Owner or Admin Personal only (TBD — see note)
Decision needed: can members create personal tasks? Your RBAC plan says "Create/assign: Member ❌" — but this was written to mean no assigning to others. Banning members from creating their own private to-do items would be quite restrictive. Recommendation: allow members to create personal (unassigned, private) tasks. Confirm before build.
Why the owner can't receive assigned tasks. The owner runs the agency, not a task queue. Having subordinates assign work to the owner breaks the authority model. The owner can still create personal tasks for themselves. This matches how tools like Asana, Linear, and Monday handle founding roles.

Admin → Admin assignment

Admins can assign tasks to other admins. Agencies often have parallel admins (Account Manager + Creative Director) who delegate work to each other. Restricting this would add unnecessary friction. Only the owner-assign-restriction is absolute.

04 Visibility Model

No visibility toggle. Visibility is implicit from assignment — cleaner UX, no confusion about who set what to private.

Personal task
assigned_to IS NULL — Visible only to the creator. Replaces the sticky-note / reminder behavior from the old system. Nobody else sees it.
Creator only
Assigned task
assigned_to IS NOT NULL — Visible to the creator, the assignee, and all owners/admins. Leadership always has full visibility over assigned work.
Creator Assignee All Owners All Admins

This is enforced at the RLS level in Postgres, not just in the UI. A member cannot query an assigned task that isn't theirs even with direct API calls.

What this means in the UI

Owner / Admin view
Sees all assigned tasks across the workspace. Personal tasks from others are invisible (by design — those are private). Has a "My Tasks" filter to narrow to tasks assigned to them.
Member view
Sees tasks assigned to them + their own personal tasks. Cannot see tasks assigned to others or tasks created by others. The "assign" UI is hidden entirely.

07 UI Approach

The existing page already has the right bones. Here's what stays and what changes:

Keep from the existing UI

Grid view with date grouping
Overdue / Upcoming / Done / Archived grouping is genuinely useful. Keep it. Add priority indicators to cards.
Kanban board
Drag-and-drop between status columns is good for team management. Extend columns from 3 to 4 to include IN_PROGRESS.
Client + campaign linking
Optional context linking to clients and campaigns is a real differentiator. Keep and display prominently on cards.
Client filter dropdown
Filter the task list to a single client. Already works well.

New additions

Assignee picker in create dialog
Owner/admin see a team member dropdown when creating a task. Members don't see it. Shows avatar + name. "Assign to myself" shortcut.
Assigned by / to display on cards
Cards show assignee avatar. If you're looking at someone else's assigned task, show "from [Creator]". Makes delegation visible at a glance.
Priority badge
Urgent = red chip, High = amber, Normal = no badge (reduces noise), Low = muted. Urgent tasks float to the top of each group.
"Assigned to me" filter
Quick filter chip to show only tasks where you are the assignee. Useful for members who only need to see their work queue.
"My personal tasks" section
Personal (unassigned) tasks render in a separate section below assigned tasks — so private to-dos don't clutter the team work queue.
Member restricted UI
For members, create button creates personal task only. Assign field is hidden. Edit/delete/archive buttons are hidden. Status toggle is shown for their assigned tasks.

Four phases. Each is independently shippable. Phases 1–2 replace the existing system; Phases 3–4 add team-specific power.

09 Open Questions Before Build

1 — Can members create personal tasks?
The RBAC plan says "Create/assign: Member ❌" but that was written before dedicated task management was scoped. Banning private to-dos is very restrictive. Recommendation: yes, allow it — members can create unassigned personal tasks, just not assign to others. Confirm this call before Phase 1.
2 — Do members need an IN_PROGRESS status?
Members can advance their assigned tasks from TODO → IN_PROGRESS → DONE. This is the main reason the status exists from a team perspective. If the answer to Q1 is yes (personal tasks for members), they'd also be able to self-advance those. Probably the right call — but confirm.
3 — Tasks in client/campaign detail pages
Currently "Go to tasks" is a nav link from the client overview. Phase 4 proposes a proper in-context task list in the client detail page. This is a worthwhile UX improvement but out of scope for the core feature. Should it be in Phase 4 or a later standalone feature? Affects scope estimate.
4 — Notifications for assigned tasks
When an owner/admin assigns a task to a member, should the member get notified? This isn't in scope for the initial build but should be designed for — particularly the edge-function pattern already used elsewhere in the codebase. Out of scope for now, but flag it as Phase 5 material.
