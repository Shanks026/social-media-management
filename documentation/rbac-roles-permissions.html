Tercero · Access Control Reference

System Roles & Permissions

Who can do what inside a single agency workspace. The system role decides access; the functional role is a cosmetic job title and never affects permissions.

Owner — runs the company Admin — runs the work Member — does assigned work

Owner

The founder/co-founder who signed up. Exactly one. Owns billing, the team, and the workspace itself.

Admin

A trusted operational deputy. Full command of client work — but nothing about the company.

Member

Delivery staff. The default for everyone who joins. Drafts work; needs approval to send it out.

Delivery staff · default on join

Many · default role

  • ✓Documents — by per-member level
  • ✕Finance, proposals, prospects, reports
  • ✕Confidential docs · create/delete clients

03

Functional roles

Cosmetic job titles — set by the joining member, never an access control.

These do not affect permissions. A Member who is a "Finance Manager" still has no finance access — access comes only from the system role above. The joining member picks their own title on the signup form. The owner can update it later via the team page. The list is a curated default; anyone can type a Custom… title for any agency type.

04

Invite flow

How a new teammate joins and gets their roles.

Step 1 — Owner

Generate an invite link

Owner picks the system role — Member or Admin — and generates a link. The link encodes system_role on the agency_invites row. No functional role is set here.

Step 2 — New member

Fill in the join form

The invitee creates their account (name, email, password) and optionally picks their own functional role / job title from a grouped list or types a custom one. The system role they're joining as is shown on the page.

Step 3 — System

Member row created

The join_team RPC reads system_role from the invite, derives permissions automatically (admin → documents:manage, member → documents:view), and inserts the agency_members row. Owner adjusts document level or promotes/demotes from the team page at any time.

05

Deliverable status flow

How a deliverable moves from draft to published — and where each role can take it.

Two approval gates, and a Member never opens one to the outside. A Member can only push work to the internal gate (Submit for approval). An Owner/Admin decides what clears it, and is the only role that can open the client gate (Send to client) or publish. Owner/Admin-authored work skips the internal gate entirely — it starts at Draft and is routed directly. The internal loop edits the same version in place; only a client-requested revision ever creates a new numbered version.
Draft Submitted Changes Requested Ready Pending Approval Needs Revision Scheduled Approved Delivered Published
Member  submits / edits Owner / Admin  approves · sends · publishes Client  approves / requests revisions

Internal workspace deliverable No client

Work for the agency's own internal account — it never leaves the workspace, so there is no client gate.

Draft Submit→ Submitted Approve→ Ready Schedule→ Scheduled → Published
↺Changes requested: Owner/Admin sends Submitted back as Changes Requested with feedback. The Member edits the same version and resubmits — no copy, no version bump.
⇝Non-social deliverable: from Ready the Owner/Admin marks it Approved → Delivered (no scheduling step).
⚡Owner/Admin authored: starts at Draft and goes straight to Scheduled / Approved — skips Submitted & Ready.

Public client deliverable Client review

Work destined for a real client — it passes the internal gate first, then the existing client review (share link).

Draft Submit→ Submitted Approve→ Ready Send to client→ Pending Approval Approves→ Scheduled → Published
↺Internal changes: the same Submitted ↔ Changes Requested loop runs between Member and Owner/Admin before anything reaches the client.
↩Client requests revisions: Pending Approval → Needs Revision spawns a new numbered version (Draft) that re-enters the gate. The only loop that versions.
⚡Owner/Admin authored: starts at Draft and sends straight to the client — skips Submitted & Ready.
